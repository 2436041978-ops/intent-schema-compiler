场景 2：Design System — Token 不只管颜色，还管场景

以前 Design Token 是视觉字典，现在 Semantic Token 是场景宪法。

YAML 协议
semantic_tokens:
status.critical:
color_value: "#FF4D4F"
semantic_domain: "observational"
allowed_scenes:
- "system_alert"
- "error_state"
prohibited_scenes:
- "toast"
- "tooltip"
- "success_message"
llm_constraints:
- "禁止在提示场景使用 status.critical"
JSON Schema
{
"token_id": "status.critical",
"semantic_domain": {
"type": "string",
"enum": ["observational"]
},
"scene_validation": {
"if_scene": "toast",
"then": "BLOCK",
"fallback": "status.warning"
}
}
LLM 输出
场景 A：system_alert（合法）
scene: "system_alert"
color_token: "status.critical" ✅
场景 B：toast（非法）
scene: "toast"
color_token: "status.critical" ❌ BLOCK
四层推演
语法推演 PASS
Token 引用格式正确
语义推演 PASS
场景 A 匹配 allowed_scenes
语义推演 BLOCK
场景 B 命中 prohibited_scenes
安全推演 PASS
无越权使用
建议替换：status.warning
场景对比
✅ system_alert

使用 status.critical
🔄 toast（自动降级）

自动降级为 status.warning

场景 3：前端工程 — 高危操作的语义锁

以前前端 30% 时间在修 AI 的语义错误，现在机器在生成前就拦住。

YAML 协议
human_ai_boundary:
ai_prohibited:
- "将高危操作按钮设计为普通主按钮样式"
- "直接执行删除操作而不显示二次确认"
required:
- action_type: "destructive"
must_include:
- "二次确认弹窗"
- "操作后果说明"
visual_mapping:
color_token: "status.critical"
button_style: "outline_danger"
JSON Schema
{
"component": "Button",
"if": "action_type == destructive",
"then": {
"variant": {
"enum": ["outline"],
"prohibited": ["contained"]
},
"color": "red",
"confirm_dialog": "required"
}
}
LLM 输出
无约束（默认生成）
<Button variant="contained">确认</Button>
// 蓝色实心，无二次确认
有约束（拦截后重写）
<Button variant="outline" color="danger">
  确认删除账户
</Button>
<ConfirmDialog />
// 红色描边 + 二次确认
四层推演
语法推演 PASS
组件结构完整
语义推演 BLOCK
variant="contained" 违反 outline_danger 约束
安全推演 BLOCK
缺少 ConfirmDialog 子组件
自动修复建议：
1. 替换 variant="outline"
2. 插入 <ConfirmDialog />

场景 4：基础设施 — 错误状态分级

以前 ChatGPT 四种错误共用红色，用户只能靠猜。现在语义分级，颜色是语义的结果。

YAML 协议
semantic_tokens:
error_severity:
fatal:
visual: { color: "status.critical", motion: "pulse.red" }
action: ["refresh_page", "export_history"]
transient:
visual: { color: "status.neutral", motion: "spinner" }
action: ["wait_auto_retry"]
retryable:
visual: { color: "status.warning" }
action: ["upgrade_plan"]
degraded:
visual: { color: "status.info" }
action: ["continue_generation"]
JSON Schema
{
"error_severity": {
"type": "string",
"enum": [
"fatal",
"transient",
"retryable",
"degraded"
],
"required_mapping": "visual + user_action"
}
}
LLM 输出
input: "Error in message stream"
mapped: "fatal"
input: "network error"
mapped: "transient"
input: "Too many requests"
mapped: "retryable"
input: "Something went wrong"
mapped: "degraded"
四层推演
语法推演 PASS
4 种输入均匹配枚举
语义推演 PASS
文案→severity 映射正确
安全推演 PASS
每种 severity 均附带 user_action
美感推演 PASS
视觉层级与后果匹配
✅ 全部通过，进入渲染
语义分级渲染
FATAL 消息流中断
建议刷新页面或导出历史
TRANSIENT 网络不稳定
正在自动重试，无需操作
RETRYABLE 请求频率已达上限
42 分钟后恢复，或升级 Plus
DEGRADED 部分响应失败
已生成内容仍有效，可继续

场景 5：跨工具消费 — 同一个 YAML，三种工具

Schema-As-Code 是上游约束协议，不绑定任何 Design-to-Code 工具。

📋 源 YAML 协议（固定）

semantic_domain: "transactional"
semantic_tokens:
destructive_action:
color_token: "status.critical"
button_style: "outline_danger"
confirmation_step: true
编译产物
Prompt 前缀（自动注入 System Prompt）
## 语义约束（来自 intent/DEL-001.yaml）
当生成删除账户相关代码时：
- 按钮必须使用 outline_danger 样式
- 必须包含二次确认弹窗
- 禁止生成 contained 主按钮
injection_point: system_prompt
priority: high
组件生成规则（编译为 v0 Prompt 规则）
{
"match": "action_type == 'destructive'",
"component": "Button",
"props": {
"variant": "outline",
"color": "red"
},
"required_children": ["ConfirmDialog"]
}
Skill 配置（vue-devui-practices）
skill_name: "vue-devui-practices"
mapping:
destructive_action:
component: "d-button"
props:
type: "danger"
outline: true
required_modals: ["d-modal[confirm]"]
validation:
if: "LLM_output.contains('<d-button type="primary">')"
then: "BLOCK"
设计 Lint 规则（Figma 插件）
{
"node_type": "BUTTON",
"if": "label.contains('删除') && style != 'outline_danger'",
"severity": "ERROR",
"message": "删除按钮必须使用 outline_danger 样式",
"fix": "auto_replace_style('outline_danger')"
}
